## 2) Two important fixes (these will block real agent discovery)

### Fix #1: Your LLM JSON parsing will fail for arrays (and your Go-Getter asks for arrays)

Your Go-Getter analysis prompt asks the model to **return a JSON array** (`[...]`).
But your model router only attempts `JSON.parse()` if the response starts with `{` (object), not `[` (array).

**Impact:** discovery can silently return a string instead of `BusinessOpportunity[]`, causing scoring/sorting to break or behave unpredictably.

**Patch idea (minimal):**

* parse when content starts with `{` **or** `[`
* strip ```json code fences
* add a “must be JSON” response_format path when the provider supports it

### Fix #2: Provider adapters are still “OpenAI-compatible only”

Your router hits `POST {baseUrl}/chat/completions` and expects an OpenAI-style response shape.
That will work for providers that intentionally mimic OpenAI’s API, but **won’t** work for providers with different endpoints/response shapes unless you add adapters.

**Recommendation:** keep your `ModelRouterService`, but implement per-provider adapters behind a single interface:

* `sendChat(messages, options) -> { text, inputTokens, outputTokens }`
* normalize responses for logging + budgets

## 6) Agent orchestration framework that fits your current platform

Right now, your app is a *control plane*. What you still need is a standardized *data plane* (“business runners”) that actually do the work and report back.

### A. The core concept: “Business Runner Contract”

Every micro-business runner must implement:

**1) Triggers**

* schedule (cron), webhook, inbox message, marketplace event, etc.

**2) Allowed Actions**

* send email? post reply? charge card? (most should require approval until stable)

**3) Budgets**

* max tokens/day, max $/day infra, max external API calls/day

**4) Telemetry (send to your platform)**
Log to `business_events`:

* `revenue` (amount)
* `cost` (amount)
* `error` (message)
* `intervention` (requiresIntervention=true)
* `agent_activity` (heartbeat, “did work”, progress)

You already support these event types and the profit aggregation logic. 

### B. Recommended runtime architecture (MVP)

**Go-Getter Control Plane (your current app)**

* catalog, scoring, API config, monitoring UI, dashboards

**Business Runner (new service, per business or multi-tenant worker)**

* queue + scheduler
* connectors (email/web/stripe/etc.)
* LLM calls via your router
* sends signed webhooks/events back to control plane

You already started sketching this with docker-compose (redis + business-agent).

### C. Orchestration components (practical + minimal)

1. **Scheduler**: cron triggers per business (every 15 min / hourly / daily)
2. **Queue**: Redis-backed (BullMQ) or Temporal later
3. **State store**: DB (business run states), plus idempotency keys
4. **Model router**: selects cheapest model per task + fallback (already exists).
5. **Budget enforcer**: if spend > limit → pause business + emit intervention event
6. **Approval workflow**: “draft output” → approve → perform irreversible action

### D. Safety/Profit guardrails (must-have)

* **Auto-pause on margin breach** (profit < 0 for N hours OR token spend spike)
* **Outbound communication rate limits**
* **Hallucination control**: constrain outputs (schemas, citations where relevant)
* **Human review** for:

  * financial transactions
  * legal docs
  * public posting (reviews/social)
  * anything that can get accounts banned
